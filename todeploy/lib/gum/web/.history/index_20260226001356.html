Mingle Platform ::: Dashboards Help


Dashboards are used to visualize in real-time the state of the devices running inside (one or more, local or remote) ExEns.
You can use Dashboards to remotely access, monitorize and even operate (to change devices states) at ExEn instances.

Dashboards has 2 operation modes:
  • Design: allows to create and modify dashboards (visualizations).
  • Execution: allows to "run" dashboards (visualizations).

Dashboards has 2 layouts:
  • Free: You place each gadget at X% and Y% position with Width% and Height% (based on browser's dimensions).
  • Grid: Uses a set of 'cards' of different sizes that can be created as pleased (when designing and also when running).

Miscellaneous:
  • Every dashboard is open in a new tab and their previews too. In this way, the initial screen is always available.
  • Dashboards are saved upon user request (diskette icon in toolbar).
  • Most part of the UI elements have a 'tooltip' that shows up when the mouse is over it.
  • Where ever it makes sense, double-click can be used.
  • While designing, the help button (at right corner inside the toolbar) shows essential help.
  • For a complete description of this tool, please refer to "The_Mingle_Standard_Platform.pdf", there is a chapter about Gum and Dashboards.
  • To close this help, click the help button again.

Security & Authentication


Gum uses a two-layer security system to protect your dashboards and API:

1. Authentication (HMAC)

Users are authenticated at login using HMAC-SHA256 cryptographic signatures.

  • Shared Secret: A secret key configured in config.json that must be known by users.
  • Configuration: Set "shared_secret" in config.json (under the "monitoring" module). If null, authentication is disabled.
  • Tolerance: Accepts timestamps within ±60 seconds by default (configurable via "hmac_tolerance").
  • Remember me: Check the "Remember me" checkbox to store the shared secret in your browser's localStorage for convenience.


2. Dashboard Access Control (Passwords)

Individual dashboards can have their own passwords for access control:

  • Master password: Grants full access to all dashboards (create, edit, rename, delete). And also to change the master password.
  • Dashboard passwords: Control which dashboards a user can view/run (creation and designing is allowed only to master password).
This is a brief explanation:
  • Only master user can change master password and dashboard passwords.
  • All dashboards are available when entering the master password.
  • Dashboards sharing the same dashboard-password (or having no password) are accessible, but only for viewing/running.
  • When running locally (LocalHost), no authentication is required and the user role is master.


Summary:

  • API Authentication: Required for all API calls when shared_secret is configured in config.json.
  • Dashboard Passwords: Optional, per-dashboard access control that works independently of API authentication.
  • Both layers can be used together for maximum security, or just one for simplicity.
  • For home automation, the shared secret provides strong security without complex setup.
